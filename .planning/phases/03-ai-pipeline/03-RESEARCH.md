## Summary

Phase 3 builds a two-step AI processing pipeline that transforms `FETCHED` articles into fully-written German-language news articles tagged to Bezirke, SEO-optimised, and guarded against cost explosion. The pipeline is powered by `@anthropic-ai/sdk` calling `claude-haiku-4-5-20251001` with structured JSON outputs — guaranteeing schema-compliant responses without custom parsing.

The Anthropic SDK (v0.80.0, March 2026) provides `client.messages.create()` with `output_config.format.type: "json_schema"` for constrained token generation, and a `messages.parse()` helper that integrates with Zod schemas for full TypeScript type safety. Every call returns a `usage` object with `input_tokens` and `output_tokens` which is aggregated in the new `PipelineRun` table — the same structure as `IngestionRun` from Phase 2 — enabling cost circuit-breaking via daily token summation.

The existing codebase provides all the scaffolding needed: DI overload pattern, `ArticleStatus` enum with `TAGGED/WRITTEN/REVIEW` already defined, `Article.isAutoGenerated = true` default, `ArticleBezirk` junction table with `taggedAt`, and `Article.metaDescription` already present in the schema. Phase 3 needs to add `seoTitle` to the Article model, add `PipelineRun` model, write `processArticles()` following `ingest()` structure, and write prompts.

**Primary recommendation:** Install `@anthropic-ai/sdk`, use `client.messages.create()` with `output_config` JSON schema mode for both pipeline steps, read token counts from `response.usage`, and mirror the `ingest()` / `IngestionRun` pattern exactly for `processArticles()` / `PipelineRun`.

### Pattern 1: Two-Step Pipeline with Structured JSON Outputs

**What:** Each article goes through Step 1 (tag) then Step 2 (write). Both steps use `client.messages.create()` with `output_config.format.type: "json_schema"`. Token counts from both calls are summed and stored in `PipelineRun`.

**When to use:** Every `FETCHED` article — mandatory for AI-01 through AI-03.

**Step 1 example:**

```typescript
// Source: platform.claude.com/docs/en/build-with-claude/structured-outputs

const Step1Schema = {
  type: 'object',
  properties: {
    bezirkSlugs: {
      type: 'array',
      items: { type: 'string' },
      description: 'Slugs of matching Bezirke from the provided list'
    },
    hasNamedPerson: {
      type: 'boolean',
      description: 'True if the article mentions a real named person'
    }
  },
  required: ['bezirkSlugs', 'hasNamedPerson'],
  additionalProperties: false

// Usage:
const response = await client.messages.create({
  model: 'claude-haiku-4-5-20251001',
  max_tokens: 256,
  system: systemPromptStep1(bezirkList),  // inject Bezirk names + gemeindeSynonyms
  messages: [{ role: 'user', content: rawArticleText }],
  output_config: {

const parsed = JSON.parse(
  (response.content.find(b => b.type === 'text') as { text: string }).text

  input: response.usage.input_tokens,
  output: response.usage.output_tokens
```

### Pattern 3: Cost Circuit-Breaker

**What:** Before processing each article (or batch), sum `total_input_tokens + total_output_tokens` across all `PipelineRun` rows where `startedAt >= start of today`. Compare to threshold. If exceeded, halt and `console.warn` structured alert.

**Implementation:**

```typescript
async function checkCostCircuitBreaker(db: PrismaClient): Promise<boolean> {
  const todayStart = new Date()
  todayStart.setHours(0, 0, 0, 0)

  const agg = await db.pipelineRun.aggregate({
    where: { startedAt: { gte: todayStart } },
    _sum: { totalInputTokens: true, totalOutputTokens: true }
  })

  const totalTokens =
    (agg._sum.totalInputTokens ?? 0) + (agg._sum.totalOutputTokens ?? 0)

  const threshold = parseInt(process.env.AI_DAILY_TOKEN_THRESHOLD ?? '500000', 10)

  if (totalTokens >= threshold) {
    console.warn(
      `[ai-pipeline] CIRCUIT_BREAKER totalTokens=${totalTokens} threshold=${threshold} — halting AI generation`
    )
    return false  // halt
  }
  return true  // proceed
}
```

**Default threshold recommendation:** `500_000` tokens/day. At $1/MTok input + $5/MTok output for Haiku 4.5, 500k mixed tokens ≈ $1–3/day for a regional news platform. Configurable via `AI_DAILY_TOKEN_THRESHOLD` env var.

### Pattern 4: Gemeinde Synonym Injection

**What:** Step 1 system prompt includes the full list of Bezirke with their `gemeindeSynonyms` to help the model match article geography to the correct Bezirk slug.

**Strategy:**

```typescript
function buildBezirkContext(bezirke: Bezirk[]): string {
  return bezirke.map(b =>
    `${b.slug} (${b.name}): ${b.gemeindeSynonyms.join(', ')}`
  ).join('\n')
}

function systemPromptStep1(bezirkContext: string): string {
  return `You are a regional news classifier for Steiermark, Austria.
Classify the provided news article into the relevant Bezirke using ONLY the slugs listed below.
Return 'steiermark-weit' if the article is state-wide and not specific to any Bezirk.
Return an empty array only if the article has no geographic relevance to Steiermark.

Available Bezirke:
${bezirkContext}

hasNamedPerson: Set to true ONLY if the article mentions a real, living, named individual person (not organisations, historical figures, or fictional characters).

Respond in the required JSON format.`
}
```

### Anti-Patterns to Avoid

- **Shared Anthropic client singleton across tests:** Create a new `Anthropic` instance per test or mock `client.messages.create` via `vi.spyOn`. Do not test against the real API in unit tests.
- **Blocking the main process on LLM calls:** `processArticles()` is called as a CLI script or by the Phase 4 scheduler — it is acceptable to await sequentially per article. No streaming needed for this use case.
- **Parsing response.content[0].text without checking block type:** Always find by `block.type === 'text'` — the response content array may contain other block types in future.
- **Storing the Anthropic API key in the database or source config:** `ANTHROPIC_API_KEY` is `.env` only — consistent with the `OTS_API_KEY` pattern established in Phase 2.
- **Re-processing TAGGED or WRITTEN articles:** `processArticles()` must query only `status: FETCHED`. Articles at higher statuses must never be reprocessed.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Guaranteed JSON output from LLM | Manual JSON prompting + regex fallbacks | `output_config.format.type: "json_schema"` | Constrained token sampling — model cannot emit invalid JSON. Manual parsing fails on model hallucinations, markdown fences, etc. |
| Zod → JSON Schema conversion | Custom schema serialiser | `z.toJSONSchema()` (Zod 3.24+) or `zodOutputFormat()` from SDK | Handles all Zod primitives, optionals, unions correctly |
| Token usage tracking | External proxy / middleware | Read from `response.usage.input_tokens` + `response.usage.output_tokens` | Already present on every API response at no extra cost |
| Named entity detection | Custom NER library (compromise, nlp.js, spacy) | Claude's own `hasNamedPerson` boolean in Step 1 | Decision is locked; Claude handles context-aware NER in German as part of Step 1 — no extra pass needed |

**Key insight:** The Anthropic SDK's structured output feature uses compiled grammar artifacts to constrain token generation at inference time. This is fundamentally different from prompt-based JSON requests — it is not possible for the model to produce schema-violating output. Do not add manual JSON validation wrappers around structured output responses.

## Common Pitfalls

### Pitfall 1: `output_config` is a newer API — SDK version matters

**What goes wrong:** Older versions of `@anthropic-ai/sdk` (pre-structured outputs GA) do not have the `output_config` parameter. TypeScript will throw a type error; or the field will be silently ignored at runtime.

**Why it happens:** Structured outputs became GA in late 2025. Earlier code examples use the tool_use JSON extraction pattern instead.

**How to avoid:** Install `@anthropic-ai/sdk@latest` (`^0.80.0`). The `output_config` field is present in the type definitions from v0.57+ (when structured outputs shipped).

**Warning signs:** TypeScript error `Property 'output_config' does not exist on type 'MessageCreateParamsNonStreaming'`.

### Pitfall 2: `messages.parse()` vs `messages.create()` confusion

**What goes wrong:** `messages.parse()` returns `response.parsed_output` (typed via Zod); `messages.create()` returns raw `Message` where content must be extracted from `response.content`. Mixing the two patterns leads to runtime errors.

**How to avoid:** Pick one pattern per step and document it. For this phase, `messages.create()` with manual `JSON.parse(textBlock.text)` is recommended — it is more explicit and easier to test with `vi.spyOn`. The `messages.parse()` / `zodOutputFormat` path is equally valid but requires the Zod helper import.

**Warning signs:** `response.parsed_output` is `undefined` when using `messages.create()`; `response.content` is not typed when using `messages.parse()`.

### Pitfall 3: `ArticleStatus` transition order not enforced

**What goes wrong:** If Step 2 writes `WRITTEN` directly without Step 1 first writing `TAGGED`, downstream queries for `TAGGED` articles return 0 rows and Phase 4 dead-man monitoring sees incorrect state.

**How to avoid:** Pipeline must write status transitions in sequence: `FETCHED → TAGGED` after Step 1, then `TAGGED → WRITTEN` (or `REVIEW`) after Step 2. Never skip the intermediate `TAGGED` state.

**Warning signs:** `listArticles({ status: 'TAGGED' })` returns 0 after a pipeline run.

### Pitfall 4: Circuit-breaker not checked before processing starts

**What goes wrong:** If the threshold is exceeded mid-batch, already-processed articles are written but remaining articles are silently skipped. If the check only happens once at batch start, a run that pushes past the threshold mid-execution cannot be stopped.

**How to avoid:** Check circuit-breaker once at the start of `processArticles()` before opening the `PipelineRun`. Log clearly how many tokens were accumulated today. For Phase 3 scope, a single pre-run check is sufficient — per-article checks add complexity not required until Phase 4 high-frequency scheduling.

**Warning signs:** Operator sees zero alert for partial batch completion.

### Pitfall 5: Prompt instability with real OTS content

**What goes wrong:** Prompts that work on synthetic test data fail on real OTS press releases. OTS content often contains legal boilerplate, source attribution lines, and embedded contact details that confuse the classifier or the rewriter.

**Why it happens:** STATE.md flags this explicitly: "Specific prompts for German-language regional news generation require iteration with real OTS content samples" — LOW confidence until validated.

**How to avoid:** Design prompts to be iterable without code changes. System prompts should be constants in dedicated files. Wave 0 stubs should test prompt structure, not prompt content. Treat prompt wording as LOW confidence until validated against real data in Phase 7.

**Warning signs:** Step 2 output articles contain "Pressemitteilung" artefacts, OTS boilerplate, or attribution text from the raw payload.

### Pitfall 6: `seoTitle` field not in current schema

**What goes wrong:** The current `prisma/schema.prisma` has `Article.metaDescription String?` but NOT `seoTitle`. Writing `seoTitle` to the database will fail with a Prisma error.

**Why it happens:** The schema was defined before Phase 3 scope was fully locked.

**How to avoid:** Phase 3 Wave 0 must include a new migration adding `Article.seoTitle String?`. The migration must be crafted manually (same approach as Phase 2's `20260321_ingestion` migration) since there is no live Postgres instance for `prisma migrate dev`.

**Warning signs:** `Prisma error: Unknown field 'seoTitle'` on `article.update()`.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `tool_use` for forced JSON extraction | `output_config.format.type: "json_schema"` | Late 2025 (GA) | Cleaner API, no need to parse tool_use content blocks |
| Manual JSON prompting with `JSON.parse` fallback | Constrained token sampling via `output_config` | Late 2025 | Model cannot emit invalid tokens — eliminates parse failures |
| Per-request token counting via `messages.countTokens` | Read `response.usage.input_tokens` / `output_tokens` post-hoc | Always available | No pre-flight token counting needed; usage is in every response |

**Deprecated/outdated:**
- `tool_use` JSON extraction pattern: still works but is the legacy approach for structured data. `output_config` is cleaner and requires less response parsing code.
- Beta header requirement for structured outputs: GA since late 2025 — no `anthropic-beta: output-schemas-2024-11-01` header needed.

## Open Questions

1. **Prompt wording for Austrian regional news**
   - What we know: Two-step structure is locked; neutral Hochdeutsch, 100–200 words is locked
   - What's unclear: Exact system prompt phrasing for journalistic tone, how to suppress OTS boilerplate, whether to include explicit word count in the prompt
   - Recommendation: Draft prompts in Wave 1 as module-level constants; treat as LOW confidence until validated against real OTS data in Phase 7

2. **`seoTitle` schema field**
   - What we know: `Article.metaDescription String?` exists; `seoTitle` does not
   - What's unclear: Whether to add a separate `seoTitle` field or repurpose the existing `title` field (current `title` is set by Phase 2 to the raw ingested title)
   - Recommendation: Add `Article.seoTitle String?` as a distinct field — `title` in the current schema is the display headline, `seoTitle` may differ for SEO purposes. Wave 0 migration required.

3. **`AI_DAILY_TOKEN_THRESHOLD` default value**
   - What we know: Haiku 4.5 pricing is $1/MTok input, $5/MTok output; a 200-word article rewrite uses roughly 400–800 input + 300–400 output tokens per two-step run
   - What's unclear: Expected daily volume at launch; operator's cost tolerance
   - Recommendation: Default to `500_000` tokens/day (≈ 400–600 articles at current token estimates; ≈ $1–3/day). Make overridable via `AI_DAILY_TOKEN_THRESHOLD` env var.
