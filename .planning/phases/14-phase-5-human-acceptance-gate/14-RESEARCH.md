## Architecture Patterns

### Phase Structure: Single Walkthrough Wave

This phase has no parallelizable work. All tests are sequential (later tests depend on a valid session from test 1). The plan should be structured as one wave with one plan.

### Walkthrough Protocol Pattern

Each test follows this shape:
1. **Pre-check** — Claude queries DB or sends HTTP request to confirm prerequisites
2. **Seed (if needed)** — Claude inserts required test data via DB tool
3. **User action** — User performs browser operation, confirms result verbally
4. **Verification** — Claude queries DB to confirm the expected state change
5. **Record** — Claude records PASS or FAIL, proceeds or investigates

### Auth Gap Test Pattern

The VERIFICATION.md anti-patterns note says `requireAuth()` was commented out. Current code (2026-03-25) shows it is active. The auth gap test should:

1. Identify the action endpoint: Next.js Server Actions are invoked via POST to the page URL with a special header (`Next-Action: <action-id>`)
2. Send `curl -X POST {APP_URL}/admin/articles -H "Next-Action: ..." -d '...'` without a session cookie
3. Expected result (given current code): `redirect('/admin/login')` — response should be a redirect, not a successful mutation
4. If the response IS a successful mutation: the fix is needed — add `requireAuth()` before the action body

**Important finding:** `requireAuth()` is already called in all Server Action wrappers (articles-actions.ts lines 220, 226, 230, 235, 240, 249, 255, 261, 267; exceptions-actions.ts lines 60, 65, 72, 78, 84; sources-actions.ts lines 172, 181, 195, 210). The vulnerability documented in 05-VERIFICATION.md was fixed in a subsequent phase (Phase 09-ad-config-wiring-auth-hardening, per STATE.md decision: "requireAuth() placed before try/catch in Server Actions"). The test should still run to confirm, but the expected outcome is that auth is already enforced.

### Seed Data Pattern

The `setup-db.ts` already exports `seedBulkArticles(prisma, count)` which inserts articles across all 13 Bezirke. For the filter test (200+ articles) this function can be adapted. Seed data for the exception queue test requires articles with `status='REVIEW'` — these must be inserted separately.

The cleanup strategy should mirror `cleanDb()` from `setup-db.ts` which shows the FK-safe deletion order:
```
pipelineRun → ingestionRun → aiSourceConfig → articleBezirk → article → bezirk → aiConfig → pipelineConfig → source
```
For targeted cleanup of seeded test data, track inserted IDs and delete by ID rather than truncating (production data must survive).

### Pipeline Trigger Pattern

The AI Config tone test requires an actual pipeline run. The pipeline entry point is:
```
bun run src/scripts/ai-run.ts
```
This runs `checkDeadMan()` → `processArticles()` → `publishArticles()`. For the test to be meaningful, there must be at least one article in `FETCHED` status (not yet AI-processed). If no FETCHED articles exist, Claude can insert one directly into the DB before triggering the pipeline.

### Sign-Off Artifact Format

`05-VERIFICATION.md` uses YAML frontmatter. The frontmatter fields to update:
- `status: human_needed` → `status: verified`
- `score: 22/23 must-haves verified` → `score: 23/23 must-haves verified` (or adjusted based on actual results)

Each `human_verification` entry should gain a `result:` field with `PASS`/`FAIL` and a brief note.

`14-SIGN-OFF.md` should follow the project's existing SUMMARY.md style — brief structured markdown with test results table, fix log (if any), and verifier signature.

## Common Pitfalls

### Pitfall 1: Outdated Auth Gap Assessment
**What goes wrong:** Treating the `requireAuth()` anti-pattern from `05-VERIFICATION.md` as still present, then reporting it as a failure when it has already been fixed.
**Why it happens:** The VERIFICATION.md anti-patterns section was written at Phase 5 verification time. Subsequent phases (Phase 09) explicitly fixed this.
**How to avoid:** Always read current source before classifying a documented issue. Current code has `requireAuth()` on every mutation. The curl test should confirm this is enforced, not discover a new vulnerability.
**Expected curl test result:** A POST without session cookie to any admin Server Action endpoint should return a redirect response (302/307 to `/admin/login`), not a 200 with mutation effects.

### Pitfall 2: Seeded Data Leaking Into Production State
**What goes wrong:** Test data inserted for the filter test or exception queue test is not cleaned up, leaving fake articles/sources in the DB after sign-off.
**Why it happens:** Cleanup is forgotten or fails partway through.
**How to avoid:** Track all seeded row IDs in variables. Use targeted deletes by ID at end of phase. Follow FK-safe deletion order from `cleanDb()`: `articleBezirk` before `article` before `bezirk`.

### Pitfall 3: AI Config Test With No FETCHED Articles
**What goes wrong:** Pipeline run is triggered but processes zero articles, so the tone change cannot be verified in actual output.
**Why it happens:** All articles may already be in PUBLISHED/REJECTED/FAILED state with nothing to process.
**How to avoid:** Before triggering the pipeline, check `SELECT COUNT(*) FROM "Article" WHERE status = 'FETCHED'`. If zero, insert a test article in FETCHED state (with realistic content) before triggering the run.

### Pitfall 4: Server Action Endpoint Discovery for Curl Test
**What goes wrong:** Claude cannot determine the Next.js action ID for a specific Server Action, making the curl exploit test impossible.
**Why it happens:** Next.js 15 Server Action IDs are generated at build time and not human-readable.
**How to avoid:** The curl test can use the simpler middleware-level check: `curl -X GET {APP_URL}/admin/articles` without a session cookie should redirect to `/admin/login`. For the Server Action POST test, the action ID can be found in the compiled `.next/server` output or the page HTML source (hidden in the form). If action ID lookup fails, document as "middleware-level auth confirmed; Server Action direct POST test skipped — action IDs not extractable without running app inspection."

### Pitfall 5: `requireAuth()` Throws NEXT_REDIRECT — Cannot Catch in Test
**What goes wrong:** Attempting to verify auth in tests by calling the Server Action directly and catching errors — NEXT_REDIRECT is not a real Error and cannot be caught in standard test patterns.
**Why it happens:** Next.js `redirect()` throws an internal NEXT_REDIRECT object, not a standard Error.
**How to avoid:** Auth verification for the human gate is done via HTTP (curl), not via vitest. This is a browser/HTTP test only.

## Open Questions

1. **Server Action endpoint IDs for curl test**
   - What we know: Next.js 15 generates opaque action IDs at build time
   - What's unclear: Whether action IDs are discoverable from running app without source inspection
   - Recommendation: Primary auth test uses GET request to `/admin/articles` (middleware-level). If page HTML is accessible, parse the `<form>` `data-action` attribute for the action ID. If not accessible, document middleware auth as confirmed and Server Action POST test as "endpoint ID not extractable without app source access."

2. **DB connection string availability at test time**
   - What we know: Plan uses `{APP_URL}` and `{DB_CONNECTION_STRING}` placeholders
   - What's unclear: Whether user will have psql, DATABASE_URL env var, or need Prisma CLI
   - Recommendation: Plan should present 3 DB tool options in priority order: (1) `psql $DATABASE_URL -c "..."`, (2) `npx prisma db execute`, (3) direct Bun/Node script. Claude picks at runtime based on what's available.

3. **ANTHROPIC_API_KEY availability for AI Config test**
   - What we know: Locked decision says "no API key fallback" — adapt gracefully
   - What's unclear: Exact adaptation path if key is unavailable
   - Recommendation: If key missing, verify DB persistence (AiConfig.tone = 'FORMAL' after save) and mark pipeline trigger as "untested — ANTHROPIC_API_KEY not available." Record test as PARTIAL-PASS with note, not FAIL.
