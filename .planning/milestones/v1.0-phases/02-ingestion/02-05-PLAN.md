<feature>
  <name>Adapter registry + core ingest() + Source DAL</name>
  <files>
    src/lib/ingestion/adapters/registry.ts,
    src/lib/ingestion/adapters/registry.test.ts,
    src/lib/ingestion/ingest.ts,
    src/lib/ingestion/ingest.test.ts,

  </files>
  <behavior>
    registry.ts:
    - Exports adapterRegistry: Partial<Record<ArticleSource, AdapterFn>>
    - OTS_AT → otsAtAdapter (or factory result), RSS → rssAdapter, MANUAL → undefined

    ingest(source: Source) → Promise<{ itemsFound: number; itemsNew: number }>:
    - Opens IngestionRun at start
    - Resolves adapter from registry; throws if none found for source.type
    - Calls adapter.fetch(source), catches errors
    - For each RawItem: computes contentHash, calls isDuplicate, skips if duplicate
    - For non-duplicates: db.article.create({ source, externalId, status: FETCHED, rawPayload, contentHash, title })
    - On success: close IngestionRun with counts, reset consecutiveFailures to 0, set lastSuccessAt=now, healthStatus=OK
    - On adapter throw: close IngestionRun with error, increment consecutiveFailures, set healthStatus DEGRADED (1-2) or DOWN (>=3)
    - Uses TypeScript overloads for DI (same as established DAL pattern)

    sources.ts DAL:
    - listSources(enabled?: boolean): returns Source[] filtered by enabled flag if provided
    - getSourceById(id): returns Source | null
    - updateSourceHealth(id, patch): updates consecutiveFailures, lastSuccessAt, healthStatus
    - All functions use TypeScript overloads for DI
  </behavior>
  <implementation>
    Health threshold constant: HEALTH_FAILURE_THRESHOLD = 3 (matches user decision: configurable default 3).
    Store as a module-level constant in ingest.ts for now; Phase 5 CMS will make it configurable per-source.
    The DEGRADED/DOWN logic: after incrementing, if consecutiveFailures >= threshold → DOWN, elif >= 1 → DEGRADED.
    Do NOT emit external alerts in Phase 2 — structured console.warn/error log entries are the alert mechanism per user decision.
  </implementation>
</feature>
