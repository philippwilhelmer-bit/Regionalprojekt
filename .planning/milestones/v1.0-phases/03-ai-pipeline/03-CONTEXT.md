# Phase 3: AI Pipeline - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Transform `FETCHED` articles into clean German-language news articles tagged to the correct Bezirk(e), SEO-optimised, flagged for review when they mention real named persons, and guarded against cost explosion — all without blocking any web request. No scheduling (Phase 4), no CMS UI (Phase 5), no reader frontend (Phase 6) — just the AI processing pipeline and its supporting data model.

</domain>

<decisions>
## Implementation Decisions

### LLM Provider
- **Anthropic Claude** — `claude-haiku-4-5-20251001` for all generation tasks
- Haiku is the default for all pipeline steps (fast, low cost, sufficient for short news rewrites)
- `ANTHROPIC_API_KEY` in `.env` only — consistent with `OTS_API_KEY` pattern from Phase 2; never in DB or committed config

### Article Output Style
- **Tone:** Neutral Hochdeutsch — objective, journalistic German; no dialect, no informal language; standard for Austrian local news
- **Length:** 100–200 words (short news brief format; suitable for mobile, lower token cost)
- **Structure:** SEO-optimised — Headline + lead paragraph + body
  - Lead paragraph (1–2 sentences) summarises the key fact and doubles as the source for the meta description
  - Body completes the story with context
  - SEO title and meta description generated as part of Step 2

### Pipeline Shape
- **Two sequential LLM calls per article:**
  - **Step 1 — Tag & Classify:** Claude receives the raw article content and returns: Bezirk tag(s), `hasNamedPerson` boolean
  - **Step 2 — Write & SEO:** Claude receives the raw content + Bezirk context and returns: rewritten article (headline + lead + body), SEO title, meta description
- **Trigger:** `processArticles()` function + `ai-run.ts` runnable CLI script — same pattern as Phase 2 `ingest()` / `ingest-run.ts`; Phase 4 scheduler calls this function on a timer
- **Each run is logged** in a `PipelineRun` table (mirrors `IngestionRun`): started_at, finished_at, articles_processed, articles_new, total_input_tokens, total_output_tokens, error — enables cost tracking and Phase 4 dead-man monitor
- **Cost circuit-breaker (AI-04):** Token-based — sum `total_input_tokens + total_output_tokens` across all PipelineRuns today, compare to a configurable daily threshold (stored in DB or env var); halt generation and emit structured alert if threshold exceeded

### Person Detection
- **Detected in Step 1** — Claude returns `hasNamedPerson: boolean` as part of the tagging call; no separate NER pass
- **When detected:** Pipeline **still completes Step 2** (full rewrite + SEO); article is stored with `status: REVIEW` instead of `WRITTEN`
- **No timeout:** REVIEW articles stay in the exception queue until an editor acts (Phase 5 CMS provides the review UI)

### Claude's Discretion
- Exact Prisma schema fields for `PipelineRun` and any new Article fields (e.g. `metaDescription`, `seoTitle`)
- Prompt engineering — system prompt wording, JSON output schema, temperature settings
- Gemeinde synonym injection strategy — how Phase 1 synonym data is passed to Step 1 to improve Bezirk tagging accuracy
- Default cost threshold value (sensible default in config/env)
- Error handling and retry logic per article within a run

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/lib/prisma.ts` — singleton PrismaClient with DI overload pattern; AI pipeline service must follow the same TypeScript overload + duck-typing pattern
- `src/lib/content/articles.ts` — DAL with status-filtered queries; pipeline uses `listArticles({ status: 'FETCHED' })` to pick up work
- `src/lib/ingestion/ingest.ts` — `IngestResult` interface + DI overload pattern + structured console logging; `processArticles()` should mirror this structure exactly
- `src/scripts/ingest-run.ts` (Phase 2 CLI) — `ai-run.ts` should follow the same per-source error aggregation pattern

### Established Patterns
- TypeScript overloads with `$connect` duck-typing for DI (required for vitest compatibility)
- `ArticleStatus` enum already in schema: `FETCHED → TAGGED → WRITTEN → REVIEW → PUBLISHED` — pipeline advances articles through these states
- `Article.isAutoGenerated: true` (default) — "Automatisch erstellt" disclosure is structural, not a separate flag to add
- `ArticleBezirk` junction with `taggedAt` — Step 1 populates this when setting `status: TAGGED`
- `Article.rawPayload: Json?` — raw source content already stored; Step 1 reads this as input
- Structured `console.warn/error` for alerting (no external services in Phase 3)

### Integration Points
- Pipeline reads `Article` rows with `status: FETCHED` — written by Phase 2 ingestion
- Pipeline writes `Article` rows advancing to `status: TAGGED → WRITTEN` (or `REVIEW` if person detected)
- `PipelineRun` table (new in Phase 3) will be read by Phase 4 dead-man monitor
- Phase 5 CMS will expose the `REVIEW` queue to editors (no Phase 3 UI needed)

</code_context>
