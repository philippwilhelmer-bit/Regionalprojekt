<feature>
  <name>processArticles() pipeline orchestrator + ai-run.ts CLI</name>
  <files>src/lib/ai/pipeline.ts, src/lib/ai/pipeline.test.ts, src/scripts/ai-run.ts</files>
  <behavior>
    processArticles(db?: PrismaClient): Promise&lt;ProcessResult&gt;
    ProcessResult: { articlesProcessed: number, articlesWritten: number, totalInputTokens: number, totalOutputTokens: number }

    Orchestration sequence:
    1. Load all Bezirke from db.bezirk.findMany() — needed for slug→id mapping and Step 1 context
    2. Check circuit-breaker via checkCostCircuitBreaker(db) — if false, log and return zero counts immediately (no PipelineRun opened)
    3. Open PipelineRun: db.pipelineRun.create({ data: { startedAt: new Date() } })
    4. Load FETCHED articles: db.article.findMany({ where: { status: 'FETCHED' } })
    5. For each FETCHED article:
       a. Extract rawPayload as article text (use rawPayload JSON stringified, or fallback to title + content)
       b. Call runStep1Tag(anthropicClient, articleText, allBezirke) → Step1Result
       c. Map bezirkSlugs to Bezirk IDs using allBezirke map
       d. Write TAGGED status + ArticleBezirk rows in db.$transaction()
       e. Call runStep2Write(anthropicClient, articleText, matchedBezirkNames) → Step2Result
       f. Write WRITTEN (or REVIEW if hasNamedPerson) + title/content/seoTitle/metaDescription
       g. Accumulate inputTokens + outputTokens from both steps
       h. Catch per-article errors: log with console.error, continue to next article
    6. Close PipelineRun with final counts and token totals
    7. Return ProcessResult

    DI pattern: zero-arg production path uses singleton PrismaClient from src/lib/prisma.ts; injected-client path used in tests. Anthropic client created once per run internally (new Anthropic() uses ANTHROPIC_API_KEY from env). For tests, mock Anthropic client is passed to the step functions — pipeline.ts passes the client as a parameter to runStep1Tag/runStep2Write to enable mocking.

    Wait — to keep testability without leaking Anthropic client into the processArticles signature, use a module-level factory that can be overridden in tests:
    ```typescript
    // Internal — exported only for test overrides
    export let _createAnthropicClient = () => new Anthropic()
    ```
    Tests can override `_createAnthropicClient` before calling processArticles(db).

    Test cases (all use createTestDb() + cleanDb() + mocked Anthropic via vi.spyOn or _createAnthropicClient override):
    - FETCHED article → after processArticles() → status is WRITTEN, bezirke tagged, title/seoTitle/metaDescription written
    - FETCHED article with hasNamedPerson=true → status is REVIEW (not WRITTEN)
    - ArticleBezirk rows created for each bezirkSlug in Step 1 result (AI-02)
    - seoTitle written to article row (SEO-02)
    - PipelineRun row created with articlesProcessed=1, totalInputTokens and totalOutputTokens summed from both steps
    - Circuit-breaker returns false → processArticles() returns { articlesProcessed:0, articlesWritten:0, ... } without opening PipelineRun
    - TAGGED/WRITTEN/REVIEW articles are NOT reprocessed (only FETCHED selected)
    - Per-article error: one article throws in Step 1 → console.error logged → other articles continue → ProcessResult reports what completed

    ai-run.ts:
    - Imports processArticles from ../lib/ai/pipeline
    - Calls processArticles() (production path, no client injection)
    - Logs result to console.log: `[ai-run] processed=${result.articlesProcessed} written=${result.articlesWritten} inputTokens=${result.totalInputTokens} outputTokens=${result.totalOutputTokens}`
    - On error: console.error + process.exit(1)
    - Guards entry: only runs main() when import.meta.main (Bun) or equivalent pattern
  </behavior>
