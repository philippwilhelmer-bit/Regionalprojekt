<feature>
  <name>Cost Circuit-Breaker</name>
  <files>src/lib/ai/circuit-breaker.ts, src/lib/ai/circuit-breaker.test.ts</files>
  <behavior>
    - checkCostCircuitBreaker(db: PrismaClient): Promise&lt;boolean&gt;
    - Returns true if daily token total is below threshold (proceed with generation)
    - Returns false if daily token total meets or exceeds threshold (halt generation)
    - Queries db.pipelineRun.aggregate with _sum on totalInputTokens and totalOutputTokens
    - WHERE clause: startedAt >= start of today (midnight local time)
    - Threshold: parseInt(process.env.AI_DAILY_TOKEN_THRESHOLD ?? '500000', 10)
    - Export AI_DEFAULT_DAILY_TOKEN_THRESHOLD = 500_000 as named constant
    - When threshold exceeded: emit console.warn with format:
      `[ai-pipeline] CIRCUIT_BREAKER totalTokens={N} threshold={N} — halting AI generation`
    - When below threshold: no console output

    Test cases (use createTestDb() + cleanDb() from src/test/setup-db.ts):
    - No PipelineRun rows today → returns true
    - PipelineRun with totalInputTokens=100, totalOutputTokens=50 today → total=150 → below 500000 → returns true
    - PipelineRun rows summing to exactly 500000 tokens → returns false (threshold met = halt)
    - PipelineRun rows summing to 600000 tokens → returns false + console.warn emitted
    - console.warn contains 'CIRCUIT_BREAKER' and the totalTokens value
    - PipelineRun from yesterday (startedAt yesterday) NOT counted — today's total = 0 → returns true
    - env var AI_DAILY_TOKEN_THRESHOLD=1000 with 1001 tokens today → returns false
    - env var not set → uses 500000 default
  </behavior>
  <implementation>
    Single exported async function accepting PrismaClient (no singleton — pipeline.ts passes the db). Use Prisma aggregate for the daily sum. Read threshold from env var with parseInt and fallback. Emit structured console.warn on breach. Return boolean.

    Note: Per RESEARCH.md pitfall 4, the circuit-breaker is checked once at the start of processArticles() before any LLM calls. This function itself does not need retry logic — it is a simple read + compare.
  </implementation>
</feature>
