## Summary

Phase 6 builds the entire public-facing reader site on top of the already-established Next.js 15 App Router infrastructure. The project is on Next.js 15.5.14 with React 19, Tailwind v4, and Prisma 6.19 — all confirmed from the installed packages. The admin area (Phase 5) established the key patterns this phase will follow: Server Components for data fetching, `await params` (params are a Promise in Next.js 15), `await searchParams`, and Tailwind v4 with `@theme inline` in globals.css instead of tailwind.config.js.

The biggest schema task is adding `publicId String @unique @default(nanoid())` to the Article model — Prisma 6 natively supports `@default(nanoid())` as confirmed by the official docs. The existing `listArticles()` DAL needs a feed-optimized variant that enforces `status: 'PUBLISHED'`, sorts pinned/featured articles to the top, and supports multiple bezirkIds. The existing `getArticlesByBezirk()` already filters by bezirkId and includes isStateWide logic.

RSS generation uses `generateRssFeed()` from the already-installed `feedsmith` package (v2.9.1) — the same package used for ingestion. This is a zero-dependency-addition path. The sitemap uses Next.js 15's native `app/sitemap.ts` file convention returning `MetadataRoute.Sitemap`. The `slugify` package is not currently installed; the recommendation is to use the `nanoid` package (already present as a transitive Prisma dependency) for publicId and write a minimal German slug function inline (urlencoded lowercased title, ~5 lines), avoiding an extra package install.

**Primary recommendation:** Build the reader homepage and article detail pages as Server Components following the established Phase 5 pattern; use a single Client Component (`BezirkModal` + `useBezirkStore` custom hook backed by localStorage) for the personalization UI; keep all data fetching server-side.

### Pattern 1: Next.js 15 Async Params (CRITICAL)
**What:** In Next.js 15, `params` and `searchParams` passed to page components are Promises and MUST be awaited.
**When to use:** Every dynamic route page and every page receiving searchParams.
**Example:**
```typescript
// Source: /node_modules/next/dist/server/request/params.d.ts
// Confirmed: params is Promise<Params> in Next.js 15

// Article detail page
type Props = { params: Promise<{ publicId: string; slug: string }> }

export default async function ArticlePage({ params }: Props) {
  const { publicId, slug } = await params
  const article = await getArticleByPublicId(publicId)
  if (!article) notFound()

  // Canonical slug enforcement: redirect if slug is wrong
  const canonical = slugify(article.seoTitle ?? article.title ?? '')
  if (slug !== canonical) {
    permanentRedirect(`/artikel/${publicId}/${canonical}`)
  }
  return <ArticleDetail article={article} />
}
```

### Pattern 2: generateMetadata for Article SEO (SEO-01)
**What:** Export async `generateMetadata` from the article page — runs server-side, cannot be in a Client Component.
**When to use:** Article detail page, Bezirk pages.
**Example:**
```typescript
// Source: /node_modules/next/dist/lib/metadata/types/metadata-interface.d.ts
import type { Metadata } from 'next'

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { publicId } = await params
  const article = await getArticleByPublicId(publicId)
  if (!article) return {}

  const baseUrl = process.env.NEXT_PUBLIC_BASE_URL ?? 'https://ennstal-aktuell.at'
  const canonicalSlug = slugify(article.seoTitle ?? article.title ?? '')

  return {
    title: article.seoTitle ?? article.title ?? 'Artikel',
    description: article.metaDescription ?? undefined,
    alternates: {
      canonical: `${baseUrl}/artikel/${article.publicId}/${canonicalSlug}`,
    },
    openGraph: {
      title: article.seoTitle ?? article.title ?? 'Artikel',
      description: article.metaDescription ?? undefined,
      url: `${baseUrl}/artikel/${article.publicId}/${canonicalSlug}`,
      type: 'article',
      publishedTime: article.publishedAt?.toISOString(),
    },
  }
}
```

### Pattern 3: Sitemap via app/sitemap.ts (SEO-03)
**What:** Next.js 15 recognizes `app/sitemap.ts` with a default export returning `MetadataRoute.Sitemap`. Serves at `/sitemap.xml` automatically.
**When to use:** Root of app directory.
**Example:**
```typescript
// Source: /node_modules/next/dist/lib/metadata/types/metadata-interface.d.ts
// MetadataRoute.Sitemap = Array<{ url: string; lastModified?: Date; changeFrequency?: ...; priority?: number }>
import type { MetadataRoute } from 'next'

export default async function sitemap(): Promise<MetadataRoute['Sitemap']> {
  const articles = await listArticles({ status: 'PUBLISHED', limit: 1000 })
  const baseUrl = process.env.NEXT_PUBLIC_BASE_URL ?? 'https://ennstal-aktuell.at'

  const articleUrls = articles.map(a => ({
    url: `${baseUrl}/artikel/${a.publicId}/${slugify(a.seoTitle ?? a.title ?? '')}`,
    lastModified: a.publishedAt ?? a.createdAt,
    changeFrequency: 'daily' as const,
    priority: 0.8,
  }))

  const bezirke = await listBezirke()
  const bezirkUrls = bezirke.map(b => ({
    url: `${baseUrl}/bezirk/${b.slug}`,
    changeFrequency: 'daily' as const,
    priority: 0.6,
  }))

  return [
    { url: baseUrl, changeFrequency: 'hourly', priority: 1.0 },
    { url: `${baseUrl}/impressum`, changeFrequency: 'monthly', priority: 0.3 },
    ...bezirkUrls,
    ...articleUrls,
  ]
}
```

### Pattern 4: RSS Route Handler (READ-06)
**What:** Next.js App Router Route Handler returning XML with correct content-type. Uses `generateRssFeed` from feedsmith (already installed).
**When to use:** `/rss/[slug]/route.ts` for per-Bezirk feeds.
**Example:**
```typescript
// Source: feedsmith generateRssFeed export confirmed in node_modules/feedsmith/dist/index.d.ts
import { generateRssFeed } from 'feedsmith'
import { getArticlesByBezirk } from '@/lib/content/articles'

export async function GET(
  _request: Request,
  { params }: { params: Promise<{ slug: string }> }
) {
  const { slug } = await params
  const articles = await getArticlesByBezirk(slug, { limit: 20 })
  const baseUrl = process.env.NEXT_PUBLIC_BASE_URL ?? 'https://ennstal-aktuell.at'

  const xml = generateRssFeed({
    channel: {
      title: `Ennstal Aktuell – ${slug}`,
      link: baseUrl,
      description: `Aktuelle Nachrichten für ${slug}`,
      items: articles.map(a => ({
        title: a.title ?? '',
        link: `${baseUrl}/artikel/${a.publicId}/${slugify(a.seoTitle ?? a.title ?? '')}`,
        description: a.content ?? '',
        pubDate: a.publishedAt ?? a.createdAt,
        guid: { value: a.publicId, isPermaLink: false },
      })),
    },
  })

  return new Response(xml, {
    headers: { 'Content-Type': 'application/rss+xml; charset=utf-8' },
  })
}
```

### Pattern 6: localStorage Bezirk Selection (READ-01)
**What:** A Client Component reads/writes localStorage for selected Bezirk IDs. Modal is controlled by a state that is initialized from localStorage on mount.
**When to use:** BezirkModal component.
**Example:**
```typescript
// 'use client'
'use client'
import { useState, useEffect } from 'react'

const STORAGE_KEY = 'selectedBezirkIds'

export function useBezirkSelection() {
  const [selected, setSelected] = useState<number[]>([])
  const [initialized, setInitialized] = useState(false)

  useEffect(() => {
    try {
      const stored = localStorage.getItem(STORAGE_KEY)
      if (stored) setSelected(JSON.parse(stored))
    } catch {}
    setInitialized(true)
  }, [])

  function save(ids: number[]) {
    setSelected(ids)
    localStorage.setItem(STORAGE_KEY, JSON.stringify(ids))
  }

  return { selected, save, initialized }
}
```

### Pattern 7: AdSense Integration (AD-01)
**What:** `next/script` with `afterInteractive` loads the AdSense SDK. A `<AdUnit>` Client Component renders `ins.adsbygoogle` elements per zone. NPA mode toggled via `data-adsbygoogle-npa="1"` when consent is rejected.
**When to use:** Root layout for script; wherever a zone appears in the UI.
**Example:**

```typescript
  crossOrigin="anonymous"

// AdUnit component ('use client'):
'use client'
export function AdUnit({ zone }: { zone: 'hero' | 'between-articles' | 'article-detail' }) {
  const slotMap: Record<string, string | undefined> = {
    'hero': process.env.NEXT_PUBLIC_ADSENSE_HERO_SLOT,
    'between-articles': process.env.NEXT_PUBLIC_ADSENSE_INLINE_SLOT,
    'article-detail': process.env.NEXT_PUBLIC_ADSENSE_ARTICLE_SLOT,

  const slot = slotMap[zone]
  if (!slot) return null
  return (
    <ins
      className="adsbygoogle"
      style={{ display: 'block' }}
      data-ad-client={process.env.NEXT_PUBLIC_ADSENSE_PUB_ID}
      data-ad-slot={slot}
      data-ad-format="auto"
      data-full-width-responsive="true"
    />
  )
}
```

### Pattern 8: Prisma Schema Migration for publicId
**What:** Add `publicId` field with `@default(nanoid())` to Article model. Prisma 6 natively supports this. Requires a new migration.
**Example migration SQL:**
```sql
-- Phase 6: publicId for reader-facing article URLs
ALTER TABLE "Article" ADD COLUMN "publicId" TEXT;
-- Backfill existing articles (generate random 10-char IDs)
UPDATE "Article" SET "publicId" = SUBSTRING(MD5(RANDOM()::TEXT), 1, 10) WHERE "publicId" IS NULL;
ALTER TABLE "Article" ALTER COLUMN "publicId" SET NOT NULL;
CREATE UNIQUE INDEX "Article_publicId_key" ON "Article"("publicId");
```
**Schema addition:**
```prisma
model Article {
  // ... existing fields ...
  publicId  String  @unique @default(nanoid(10))
}
```

### Anti-Patterns to Avoid
- **Sync localStorage read at render:** Reading localStorage synchronously during SSR causes hydration mismatches. Always read in `useEffect`.
- **`use client` on data-fetching components:** The article list page should be a Server Component that passes data to a Client Component wrapper only for interactivity (infinite scroll trigger, Bezirk filter).
- **`params` without await:** In Next.js 15, `params` is a Promise. Destructuring without `await` produces a deprecation warning and will fail in future versions.
- **Generating publicId in application code instead of Prisma:** Using `@default(nanoid(10))` in the schema means Prisma generates it on every `article.create()` automatically, including backfills via migration.

## Common Pitfalls

### Pitfall 1: Tailwind v4 Class Discovery
**What goes wrong:** Tailwind v4 scans source files for class names — dynamically constructing class strings like `bg-${bezirkColor}-500` will not generate the CSS class.
**Why it happens:** Tailwind v4 (like v3) statically analyzes source files; it cannot see runtime-interpolated class names.
**How to avoid:** Define a static color map for all 13 Bezirke as a constant object mapping slug → full Tailwind class name (e.g., `{ liezen: 'from-emerald-600 to-emerald-800' }`). Use the full class string, never interpolated fragments.
**Warning signs:** Gradient placeholder renders with no color in production build.

### Pitfall 2: Hydration Mismatch from localStorage
**What goes wrong:** Server renders "Alle Nachrichten" (no selection); client reads localStorage and switches to "Mein Bezirk" — React throws hydration mismatch error.
**Why it happens:** localStorage is browser-only; SSR renders with no selection.
**How to avoid:** The feed heading and modal visibility state must only be set after `useEffect` fires (`initialized` flag pattern). Server renders the default state. No suppression hacks.
**Warning signs:** React console error "Text content did not match" or "Hydration failed".

### Pitfall 3: Canonical Slug Redirect Loop
**What goes wrong:** If the slug is empty or the article has no title, slugify returns an empty string, and the redirect target matches the current URL, causing an infinite redirect.
**Why it happens:** `permanentRedirect` fires whenever `slug !== canonical`, and if canonical is empty, every URL will redirect to `/artikel/[publicId]/`.
**How to avoid:** Treat empty slug as valid fallback; only redirect if `canonical !== '' && slug !== canonical`.
**Warning signs:** Next.js NEXT_REDIRECT error in logs.

### Pitfall 4: getArticlesByBezirk Returns FETCHED/WRITTEN Articles
**What goes wrong:** The existing `getArticlesByBezirk()` does not filter by `status`. Reader-facing endpoints will surface draft/pipeline articles.
**Why it happens:** The DAL was written for RSS ingestion context, not reader-facing use.
**How to avoid:** All reader-facing DAL calls MUST include `status: 'PUBLISHED'` filter. Extend `getArticlesByBezirk()` to accept a status option, or create `getPublishedArticlesByBezirk()` wrapper.
**Warning signs:** Unpublished article titles appearing in the reader feed during testing.

### Pitfall 5: AdSense NPA Mode Timing
**What goes wrong:** AdSense script executes before the cookie consent localStorage value is read, rendering personalized ads even when consent is rejected.
**Why it happens:** `afterInteractive` script fires early in page lifecycle; consent state may not be read yet.
**How to avoid:** Set `window.adsbygoogle = window.adsbygoogle || []` and push `{ params: { google_npa: 1 } }` before the adsbygoogle.js script fires, based on localStorage value read synchronously (localStorage IS synchronous — only `window` is undefined during SSR, not in the browser). Use an inline `beforeInteractive` script or `<script dangerouslySetInnerHTML>` that reads consent and sets NPA mode before AdSense loads.
**Warning signs:** GDPR audit tool reports personalized ads served without consent.

### Pitfall 6: RSS Feed CDATA for HTML Content
**What goes wrong:** Article body contains HTML or special characters that break RSS XML validity.
**Why it happens:** `content` field may contain `<`, `>`, `&` from the AI-generated text.
**How to avoid:** feedsmith's `generateRssFeed` handles CDATA wrapping automatically when `description` is provided. Verify article content is stored as plain text or that feedsmith wraps it.
**Warning signs:** RSS validator reports malformed XML; feed readers display raw tags.

### Pitfall 7: Feed Sort Order for Pinned/Featured Articles
**What goes wrong:** `listArticles()` currently sorts only by `publishedAt DESC, createdAt DESC`. Pinned articles must appear first regardless of date.
**Why it happens:** The existing DAL sort does not account for `isPinned` or `isFeatured`.
**How to avoid:** The reader variant of `listArticles` must use `orderBy: [{ isPinned: 'desc' }, { isFeatured: 'desc' }, { publishedAt: 'desc' }]`. Do not rely on post-fetch sort in JavaScript — let the DB sort.
**Warning signs:** Pinned articles appear in chronological position rather than at the top.

## Open Questions

1. **feedsmith generateRssFeed input shape**
   - What we know: Type signature is `XmlGenerateMain<Rss.Feed<Date, Rss.PersonLike>, DeepPartial<Rss.Feed<DateLike, Rss.PersonLike>>>`. The `RssFeed` type from the ingestion side uses `item.guid.value`.
   - What's unclear: The exact required fields for `generateRssFeed` input — `channel.title`, `channel.link`, `channel.description`, `channel.items` are expected but the exact nested type needs verification at implementation.
   - Recommendation: During Wave 0 test setup, write a unit test that calls `generateRssFeed` with a minimal fixture and validates the output is valid XML with expected fields. This is LOW risk since feedsmith is in active use for ingestion.

2. **Austrian MedienG/ECG exact field requirements**
   - What we know: §25 MedienG requires Medieninhaber, Unternehmensgegenstand, Blattlinie. ECG §5 requires name, address, contact. DSGVO requires privacy policy.
   - What's unclear: Whether the AI disclosure section satisfies any Austrian regulatory requirement or is purely a voluntary best practice. Noted as a blocker concern in STATE.md.
   - Recommendation: Claude drafts the Impressum as a template with clearly labeled `[PLACEHOLDER]` fields and a `<!-- TODO: legal review before launch -->` comment. The planner should include this as an explicit operator task, not an automated test.

3. **Infinite scroll implementation approach**
   - What we know: `listArticles()` already supports `limit` and `offset`. IntersectionObserver API is browser-native.
   - What's unclear: Whether to use a Server Action for loading more articles (cleaner) or a `/api/articles` route handler. Server Actions are the Phase 5 pattern.
   - Recommendation: Use a Server Action `loadMoreArticlesAction(bezirkIds, offset)` called from the Client Component scroll trigger. Consistent with established Phase 5 pattern.
