# Phase 5: Editorial CMS - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

A single authenticated admin interface at `/admin` where editors manage the full content lifecycle: write and publish manual articles, curate automated content (pin, feature, edit, soft-delete), review and approve/reject flagged exception-queue articles, configure RSS/OTS sources, and set global AI generation behaviour with per-source overrides — all without touching code or config files.

</domain>

<decisions>
## Implementation Decisions

### Editor authentication
- **Mechanism:** Custom HTML login form at `/admin/login` — sets a signed httpOnly session cookie on success
- **Credentials:** Single `ADMIN_PASSWORD` env var (and optionally `ADMIN_EMAIL`). One editor, no DB dependency for auth.
- **Guard:** Next.js middleware intercepts all `/admin/**` routes (except `/admin/login`); checks session cookie; redirects to login if absent or invalid
- **Location:** `/admin` as a route group inside the existing Next.js app — shares the same Prisma client, DB, and deploy

### Admin UI structure
- **Navigation:** Persistent left sidebar with four sections: Articles, Exception Queue, Sources, AI Config
- **Sections:**
  - **Articles** — filterable list (Bezirk, source, status, date range); inline edit, pin, feature, soft-delete; create manual article button
  - **Exception Queue** — REVIEW-status articles only; article + source text side-by-side; Approve (→ PUBLISHED) or Reject (→ REJECTED) in one click
  - **Sources** — source list with health dashboard; add/edit/disable sources
  - **AI Config** — global settings + per-source overrides; editable without deploy

### Manual article authoring (CMS-01)
- Editor fills in: title, content body, Bezirk(e) (multi-select), seoTitle, metaDescription, pin toggle, feature toggle
- Source set to `MANUAL`, `isAutoGenerated = false`
- Status goes directly to `PUBLISHED` on save (no queue, no AI step)
- Bezirke manually assigned by editor — AI tagging does NOT run on MANUAL articles

### Article curation (CMS-02)
- **Pin / Feature:** Boolean toggles on Article row (new `isPinned` and `isFeatured` fields — Phase 6 reader will use these for homepage ordering)
- **Edit:** Editor can update title, content, seoTitle, metaDescription, Bezirke on any article regardless of status
- **Soft-delete:** Sets `Article.status = REJECTED` — reuses existing enum value; reader site filters these out; reversible

### Exception queue review (CMS-03)
- **Approve:** Sets `Article.status = PUBLISHED` immediately (one click, no intermediate state)
- **Reject:** Sets `Article.status = REJECTED`
- **Side-by-side view:** Rewritten article (content field) displayed alongside raw source text (rawPayload field)

### AI config storage (AICONF-01, AICONF-02, AICONF-03)
- **New `AiConfig` table** — singleton row (always exactly one row); global defaults:
  - `tone` enum: `NEUTRAL` / `FORMAL` / `CONVERSATIONAL` (default: `NEUTRAL`)
  - `articleLength` enum: `SHORT` (100–150w) / `MEDIUM` (150–250w) / `LONG` (250–400w) (default: `MEDIUM`)
  - `styleNotes` String? — freeform instruction appended to system prompt (e.g. "Erwähne immer den Bezirk im ersten Satz")
  - `modelOverride` String? — NULL = use default (claude-haiku-4-5-20251001); set to "sonnet" for higher quality
- **New `AiSourceConfig` table** — per-source overrides; linked to `Source` by sourceId; any subset of AiConfig fields; NULL fields = fall through to global
- **Pipeline runtime:** `processArticles()` reads `AiConfig` row (and any `AiSourceConfig` rows) at the start of each run — changes take effect on next scheduled cycle without restart
- Supersedes the hardcoded `SYSTEM_PROMPT_TEMPLATE` constants in `step1-tag.ts` and `step2-write.ts` — these read from DB config instead

### Source management (CMS-04)
- **Add source:** URL + type (OTS_AT / RSS) — creates new Source row
- **Edit source:** poll interval (minutes), health failure threshold (new `healthFailureThreshold` Int field on Source; default 3 — supersedes the hardcoded `HEALTH_FAILURE_THRESHOLD = 3` constant from ingest.ts), enabled toggle
- **Source list shows per source:**
  - Health status badge (OK / DEGRADED / DOWN) — from `Source.healthStatus`
  - Last successful poll timestamp — from `Source.lastSuccessAt`
  - Recent ingestion run stats (items found / items new) — from latest `IngestionRun` row
  - FAILED + ERROR article count — count of `Article` rows with this source in those statuses

### Pipeline configuration (from Phase 4 decisions)
- **New `PipelineConfig` table** — singleton row with global pipeline tunables:
  - `maxRetryCount` Int (default 3) — supersedes hardcoded `MAX_RETRY_COUNT = 3` constant in pipeline.ts
  - `deadManThresholdHours` Int (default 6) — supersedes hardcoded silence window in dead-man.ts
- Pipeline reads `PipelineConfig` at the top of each run (same pattern as AiConfig)
- PipelineConfig editable from admin UI (in AI Config section or a dedicated Pipeline Config section — Claude's discretion)

### Claude's Discretion
- Exact sidebar styling and responsive/mobile behaviour
- Loading states, empty states, and pagination in the article list
- Form validation UX (client-side vs server-side)
- Toast/notification pattern for save confirmations
- Whether PipelineConfig lives as its own admin section or is appended to the AI Config page
- Exact Prisma migration strategy for new fields (`isPinned`, `isFeatured`, `healthFailureThreshold`) and new tables (`AiConfig`, `AiSourceConfig`, `PipelineConfig`)
- Session cookie signing approach (ADMIN_SESSION_SECRET env var + iron-session or similar)

</decisions>

<specifics>
## Specific Ideas

- Exception queue: side-by-side view of `Article.content` (rewritten) and `Article.rawPayload` (original) — editor can compare before approving
- Source health dashboard should show all four data points at a glance (health badge, last success, run stats, failed article count) — editor should be able to diagnose a broken source without navigating away
- The `ADMIN_PASSWORD` guard is intentionally minimal — "this is a small team / single-editor tool, not a multi-tenant SaaS"
- Manual articles bypass AI entirely — no Step 1, no Step 2, no PipelineRun record created

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/lib/content/articles.ts` — DAL with TypeScript overload DI pattern; CMS Server Actions should use the same pattern (injected client for tests, singleton for production)
- `src/lib/content/sources.ts` — `listSources()`, existing source CRUD pattern to extend for CMS source management
- `src/lib/prisma.ts` — singleton PrismaClient with duck-typed DI; all new admin services follow this pattern
- `ArticleStatus` enum already has `REVIEW`, `PUBLISHED`, `REJECTED`, `FAILED`, `ERROR` — no new statuses needed for Phase 5
- `SourceHealth` enum already has `OK`, `DEGRADED`, `DOWN`
- `Source` model already has `pollIntervalMinutes`, `enabled`, `healthStatus`, `lastSuccessAt`, `consecutiveFailures`, `runs` relation

### Established Patterns
- TypeScript overloads + duck-typing `$connect` for DI (required for vitest compatibility) — all new services must follow this
- Structured `console.warn` for alerting — `{ type, ...context }` shape
- `import.meta.url` guard for runnable scripts
- Singleton DB row pattern: `AiConfig` and `PipelineConfig` follow the "find first or create" approach (similar to how PipelineRun works)
- AGENTS.md note: this version of Next.js may have breaking changes — read `node_modules/next/dist/docs/` before writing any admin route/Server Action code

### Integration Points
- `pipeline.ts` `processArticles()` must read `AiConfig` + `AiSourceConfig` at run start (replace hardcoded constants)
- `ingest.ts` must read `Source.healthFailureThreshold` instead of the module-level `HEALTH_FAILURE_THRESHOLD` constant
- `pipeline.ts` must read `PipelineConfig.maxRetryCount` instead of `MAX_RETRY_COUNT` constant
- `dead-man.ts` must read `PipelineConfig.deadManThresholdHours` instead of hardcoded 6-hour window
- Phase 6 reader frontend will use `isPinned` and `isFeatured` fields added in Phase 5
