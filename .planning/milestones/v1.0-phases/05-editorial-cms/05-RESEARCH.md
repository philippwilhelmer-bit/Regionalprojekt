### Locked Decisions

**Editor authentication**
- Mechanism: Custom HTML login form at `/admin/login` — sets a signed httpOnly session cookie on success
- Credentials: Single `ADMIN_PASSWORD` env var (and optionally `ADMIN_EMAIL`). One editor, no DB dependency for auth.
- Guard: Next.js middleware intercepts all `/admin/**` routes (except `/admin/login`); checks session cookie; redirects to login if absent or invalid
- Location: `/admin` as a route group inside the existing Next.js app — shares the same Prisma client, DB, and deploy

**Admin UI structure**
- Navigation: Persistent left sidebar with four sections: Articles, Exception Queue, Sources, AI Config
- Articles — filterable list (Bezirk, source, status, date range); inline edit, pin, feature, soft-delete; create manual article button
- Exception Queue — REVIEW-status articles only; article + source text side-by-side; Approve (→ PUBLISHED) or Reject (→ REJECTED) in one click
- Sources — source list with health dashboard; add/edit/disable sources
- AI Config — global settings + per-source overrides; editable without deploy

**Manual article authoring (CMS-01)**
- Editor fills in: title, content body, Bezirk(e) (multi-select), seoTitle, metaDescription, pin toggle, feature toggle
- Source set to `MANUAL`, `isAutoGenerated = false`
- Status goes directly to `PUBLISHED` on save (no queue, no AI step)
- Bezirke manually assigned by editor — AI tagging does NOT run on MANUAL articles

**Article curation (CMS-02)**
- Pin / Feature: Boolean toggles on Article row (new `isPinned` and `isFeatured` fields — Phase 6 reader will use these for homepage ordering)
- Edit: Editor can update title, content, seoTitle, metaDescription, Bezirke on any article regardless of status
- Soft-delete: Sets `Article.status = REJECTED` — reuses existing enum value; reader site filters these out; reversible

**Exception queue review (CMS-03)**
- Approve: Sets `Article.status = PUBLISHED` immediately (one click, no intermediate state)
- Reject: Sets `Article.status = REJECTED`
- Side-by-side view: Rewritten article (content field) displayed alongside raw source text (rawPayload field)

**AI config storage (AICONF-01, AICONF-02, AICONF-03)**
- New `AiConfig` table — singleton row; global defaults: `tone` enum (NEUTRAL/FORMAL/CONVERSATIONAL), `articleLength` enum (SHORT/MEDIUM/LONG), `styleNotes` String?, `modelOverride` String?
- New `AiSourceConfig` table — per-source overrides linked to `Source` by sourceId; NULL fields fall through to global
- Pipeline runtime reads `AiConfig` + `AiSourceConfig` at run start — changes take effect on next cycle without restart
- Supersedes hardcoded `SYSTEM_PROMPT_TEMPLATE` in step1-tag.ts and step2-write.ts

**Source management (CMS-04)**
- Add source: URL + type (OTS_AT / RSS) — creates new Source row
- Edit source: poll interval (minutes), `healthFailureThreshold` (new Int field on Source; default 3), enabled toggle
- Source list shows per source: health status badge, last successful poll, recent run stats, FAILED + ERROR article count

**Pipeline configuration (from Phase 4 decisions)**

- Pipeline reads `PipelineConfig` at top of each run
- PipelineConfig editable from admin UI (AI Config section or dedicated section — Claude's discretion)

## Summary

Phase 5 builds an authenticated admin interface at `/admin` inside the existing Next.js 15 App Router project. All decisions are locked — this is pure implementation research. The interface is a route group inside `src/app/(admin)/` sharing the existing Prisma client and DB. Authentication uses a signed httpOnly cookie verified by Next.js middleware.

The biggest technical risk is Next.js 15's **async cookies/headers APIs** — `cookies()` and `headers()` now return `Promise` (verified against installed `node_modules/next/dist/server/request/cookies.d.ts`). Any code that calls these synchronously will break silently in some contexts or throw at runtime. All Server Actions and middleware must `await cookies()` before reading values.

The second significant area is the **Prisma schema migration**: four schema changes are needed (two new fields on existing models, three new singleton tables). The project uses manually crafted SQL migrations applied via pgLite in tests, so migration SQL must be written by hand alongside schema changes. The singleton "find-first-or-create" pattern is already used by PipelineRun and must be replicated for AiConfig and PipelineConfig.

**Primary recommendation:** Build in wave order — schema + DAL first (Wave 1), then auth middleware (Wave 2), then Server Actions (Wave 3), then UI pages (Wave 4) — keeping each wave testable before the next begins.

### Pattern 1: Route Group for Admin
**What:** `(admin)` route group in `src/app/` creates no URL segment but allows a separate layout. All pages under it share the admin shell layout without affecting the public site.
**When to use:** When admin and public routes need different layouts within the same Next.js app.

```typescript
// src/app/(admin)/layout.tsx — server component
import { cookies } from 'next/headers'
import { verifySessionCookie } from '@/lib/admin/auth'
import { redirect } from 'next/navigation'

export default async function AdminLayout({ children }: { children: React.ReactNode }) {
  // Skip auth check for login page
  // (middleware handles redirect — layout is a secondary guard)
  const cookieStore = await cookies()  // MUST await in Next.js 15
  const session = cookieStore.get('admin_session')
  if (!session || !verifySessionCookie(session.value)) {
    redirect('/admin/login')
  }
  return (
    <div className="flex h-screen">
      <Sidebar />
      <main className="flex-1 overflow-auto p-6">{children}</main>
    </div>
  )
}
```
Source: verified against `node_modules/next/dist/server/request/cookies.d.ts` — `cookies()` returns `Promise<ReadonlyRequestCookies>`

### Pattern 2: Next.js Middleware for Auth Guard
**What:** `middleware.ts` at project root intercepts all `/admin/**` requests except `/admin/login`. Reads the session cookie, verifies the HMAC signature, and redirects to login if invalid.
**When to use:** Stateless per-request auth check before any rendering.

```typescript
// middleware.ts (project root)
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl
  if (pathname === '/admin/login') return NextResponse.next()

  const session = request.cookies.get('admin_session')
  if (!session || !verifySessionCookieEdge(session.value)) {
    return NextResponse.redirect(new URL('/admin/login', request.url))
  }
  return NextResponse.next()
}

export const config = {
  matcher: ['/admin/:path*'],
}
```

**Important:** Middleware runs in the Edge runtime. Node `crypto` is NOT available. Use the Web Crypto API (`crypto.subtle`) for HMAC verification in middleware, OR use a symmetric approach (e.g., check a known HMAC value computed with `SubtleCrypto`). Alternatively, keep the cookie as a random opaque token stored nowhere (pure signed value) and verify with Web Crypto.

Source: verified against `node_modules/next/dist/server/web/exports/index.d.ts` — middleware gets `NextRequest`, not Node `IncomingMessage`.

### Pattern 3: Server Actions (Next.js 15)
**What:** `'use server'` functions called from Client Components via form actions or imperative calls. Mutations go through Server Actions — never raw API routes — to keep auth and validation in one layer.
**When to use:** All CMS write operations (create, update, delete, approve, reject).

```typescript
// src/lib/admin/articles-actions.ts
'use server'
import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'
import { z } from 'zod'
import { verifySessionCookie } from './auth'
import { prisma } from '../prisma'

async function requireAuth() {
  const cookieStore = await cookies()  // await required in Next.js 15
  const session = cookieStore.get('admin_session')
  if (!session || !verifySessionCookie(session.value)) {
    redirect('/admin/login')
  }
}

const CreateArticleInput = z.object({
  title: z.string().min(1),
  content: z.string().min(1),
  bezirkIds: z.array(z.number()),
  seoTitle: z.string().optional(),
  metaDescription: z.string().optional(),
  isPinned: z.boolean().default(false),
  isFeatured: z.boolean().default(false),
})

export async function createManualArticle(formData: FormData) {
  await requireAuth()
  const input = CreateArticleInput.parse({
    title: formData.get('title'),
    // ...
  })
  const article = await prisma.article.create({
    data: {
      source: 'MANUAL',
      isAutoGenerated: false,
      status: 'PUBLISHED',
      publishedAt: new Date(),
      isPinned: input.isPinned,
      isFeatured: input.isFeatured,
      title: input.title,
      content: input.content,
      seoTitle: input.seoTitle,
      metaDescription: input.metaDescription,
    },
  })
  // Create ArticleBezirk junction rows
  await prisma.articleBezirk.createMany({
    data: input.bezirkIds.map(bezirkId => ({ articleId: article.id, bezirkId })),
  })
}
```
Source: verified against `node_modules/next/dist/server/app-render/action-handler.d.ts`

### Pattern 5: DAL DI overload (must match established pattern)
All new DAL functions and Server Action DB calls must follow the existing duck-typed overload pattern from `src/lib/content/articles.ts` — zero-arg production path uses singleton, injected client path used in tests. Check `'$connect' in clientOrOptions` for duck-typing.

### Anti-Patterns to Avoid
- **Synchronous `cookies()` call:** `cookies()` returns `Promise` in Next.js 15 — calling `.get()` without `await` gives stale/undefined values at runtime. Always `const cookieStore = await cookies()`.
- **Using Node `crypto` in middleware:** Middleware runs in Edge runtime. Use `crypto.subtle` (Web Crypto API) for HMAC in middleware. Node `crypto` is fine in Server Actions (Node runtime).
- **Direct `prisma.article.update()` in pages:** All mutations go through Server Actions, not page-level code.
- **New ArticleStatus values for admin actions:** The existing enum covers all states. Adding values would require migration and breaks the pipeline's enum expectations.
- **Forgetting to `await` cookies before reading in layout:** Layout auth check is a secondary guard; middleware is primary. Both must `await cookies()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session cookie HMAC | Custom crypto wrapper | Node `crypto.createHmac('sha256', secret)` in Server Actions; `crypto.subtle` in middleware | Both are built-in, well-tested, zero-dep |
| Form validation | Ad-hoc null checks | Zod (already installed) | Declarative, composable, TypeScript inference |
| DB singleton row | Bespoke init script | `findFirst()` with create-on-null in DAL | Idempotent, no migration state needed |
| Filter query building | String concatenation | Prisma `where` object composition | Type-safe, injection-proof |
| Pagination | Manual OFFSET/LIMIT strings | Prisma `take`/`skip` | Existing pattern in `listArticles()` |

**Key insight:** The entire admin layer is Server Components + Server Actions — no client-side data fetching, no fetch() calls, no API routes needed for mutations. Prisma is called directly from Server Actions.

## Common Pitfalls

### Pitfall 1: `cookies()` is async in Next.js 15
**What goes wrong:** Code written for Next.js 13/14 calls `cookies().get('name')` synchronously. In Next.js 15 this returns the underlying object via deprecated shim but emits deprecation warnings and breaks in strict mode / future versions.
**Why it happens:** Breaking change in Next.js 15 — `cookies()`, `headers()`, `params`, `searchParams` all return Promises.
**How to avoid:** Always `const cookieStore = await cookies()` then `cookieStore.get(...)`.
**Warning signs:** TypeScript type `UnsafeUnwrappedCookies` appearing in type errors; `@next-codemod-error` comments in generated code.

Source: verified in `node_modules/next/dist/server/request/cookies.d.ts` — function signature is `declare function cookies(): Promise<ReadonlyRequestCookies>`

### Pitfall 2: Node `crypto` unavailable in middleware Edge runtime
**What goes wrong:** Using `import { createHmac } from 'node:crypto'` in `middleware.ts` throws at build time or runtime — middleware runs in the Edge runtime which has Web API subset only.
**Why it happens:** Middleware is deployed to Edge (Vercel Edge, Cloudflare Workers) by default — no Node.js APIs.
**How to avoid:** In `middleware.ts`, use `crypto.subtle.verify()` with `SubtleCrypto` HMAC. In Server Actions (Node runtime), `node:crypto` is fine.
**Warning signs:** Build error "crypto is not defined" or "Module not found: Can't resolve 'node:crypto'" in middleware context.

### Pitfall 3: rawPayload is Json? — needs type assertion for display
**What goes wrong:** `Article.rawPayload` is `Json?` in Prisma — TypeScript type is `JsonValue | null`. Rendering it requires `JSON.stringify()` or a typed cast to the expected shape.
**Why it happens:** Prisma Json fields have no compile-time shape.
**How to avoid:** Define a `RawPayload` type (matching actual OTS/RSS shapes) and cast with `as unknown as RawPayload`. Add a fallback to `JSON.stringify(rawPayload, null, 2)` for unknown shapes.

### Pitfall 4: Migration must include ALL Phase 5 schema changes
**What goes wrong:** Applying schema changes to `schema.prisma` without creating the corresponding migration SQL. The test `createTestDb()` loads migrations from `prisma/migrations/` — if migration SQL is missing, tests run against an outdated schema.
**Why it happens:** Project uses manually crafted migration SQL (no live DB for `prisma migrate dev`).
**How to avoid:** Create one migration directory (e.g. `20260322_phase5`) with a single `migration.sql` covering: `isPinned`, `isFeatured`, `Source.healthFailureThreshold`, `AiConfig` table, `AiSourceConfig` table, `PipelineConfig` table. Add new singleton tables to `cleanDb()` in `setup-db.ts`.

### Pitfall 5: PipelineConfig / AiConfig singleton — must update `cleanDb()`
**What goes wrong:** New singleton tables are not added to `cleanDb()` — test runs accumulate singleton rows across tests, causing "unique constraint" errors or stale config reads.
**How to avoid:** Add `prisma.aiConfig.deleteMany()`, `prisma.aiSourceConfig.deleteMany()`, `prisma.pipelineConfig.deleteMany()` to `cleanDb()` in the correct order (AiSourceConfig before AiConfig, AiConfig and PipelineConfig have no FK dependencies).

### Pitfall 6: Server Action `redirect()` throws — this is intentional
**What goes wrong:** Wrapping a Server Action in `try/catch` and catching the redirect throws an error.
**Why it happens:** `redirect()` in Next.js Server Actions throws a special `NEXT_REDIRECT` error internally — catching it with a generic `catch(e)` swallows the redirect.
**How to avoid:** Do not wrap `redirect()` calls in `try/catch`. Perform validation before redirect, then call redirect outside any try block.

### Pitfall 7: Article filter for exception queue must exclude FAILED/REJECTED
**What goes wrong:** Exception queue query uses `status: 'REVIEW'` — correct. But a wide `listArticles()` with no filter would show FAILED/ERROR/REJECTED articles in the main list, confusing the editor.
**How to avoid:** Admin article list default filter: `status IN ('FETCHED', 'TAGGED', 'WRITTEN', 'REVIEW', 'PUBLISHED')` — REJECTED and FAILED excluded by default, with an explicit toggle to show them.

## Code Examples

### Session cookie — sign and verify (Node runtime / Server Actions)
```typescript
// src/lib/admin/auth.ts
import { createHmac, timingSafeEqual } from 'node:crypto'

const SESSION_COOKIE_NAME = 'admin_session'

function getSecret(): string {
  const s = process.env.ADMIN_SESSION_SECRET
  if (!s) throw new Error('ADMIN_SESSION_SECRET env var is not set')
  return s
}

export function signSessionCookie(value: string): string {
  const hmac = createHmac('sha256', getSecret()).update(value).digest('hex')
  return `${value}.${hmac}`
}

export function verifySessionCookie(cookie: string): boolean {
  const lastDot = cookie.lastIndexOf('.')
  if (lastDot === -1) return false
  const value = cookie.slice(0, lastDot)
  const sig = cookie.slice(lastDot + 1)
  const expected = createHmac('sha256', getSecret()).update(value).digest('hex')
  try {
    return timingSafeEqual(Buffer.from(sig, 'hex'), Buffer.from(expected, 'hex'))
  } catch {
    return false
  }
}
```

### Session cookie — verify in Edge middleware (Web Crypto)
```typescript
// middleware.ts — Edge runtime, must use SubtleCrypto
async function verifySessionCookieEdge(cookie: string, secret: string): Promise<boolean> {
  const lastDot = cookie.lastIndexOf('.')
  if (lastDot === -1) return false
  const value = cookie.slice(0, lastDot)
  const sig = cookie.slice(lastDot + 1)
  const enc = new TextEncoder()
  const key = await crypto.subtle.importKey(
    'raw', enc.encode(secret), { name: 'HMAC', hash: 'SHA-256' }, false, ['verify']
  )
  const sigBytes = Uint8Array.from(sig.match(/.{2}/g)!.map(b => parseInt(b, 16)))
  return crypto.subtle.verify('HMAC', key, sigBytes, enc.encode(value))
}
```

### listArticles extended for admin filters
```typescript
// Extension of existing DAL to support admin multi-filter
import type { ArticleStatus, ArticleSource, Prisma } from '@prisma/client'

export async function listArticlesAdmin(db: PrismaClient, opts: {
  bezirkId?: number
  sourceType?: ArticleSource
  status?: ArticleStatus | ArticleStatus[]
  fromDate?: Date
  toDate?: Date
  limit?: number
  offset?: number
}) {
  const where: Prisma.ArticleWhereInput = {}
  if (opts.bezirkId) where.bezirke = { some: { bezirkId: opts.bezirkId } }
  if (opts.sourceType) where.source = opts.sourceType
  if (opts.status) where.status = Array.isArray(opts.status) ? { in: opts.status } : opts.status
  if (opts.fromDate || opts.toDate) {
    where.createdAt = {
      ...(opts.fromDate ? { gte: opts.fromDate } : {}),
      ...(opts.toDate ? { lte: opts.toDate } : {}),
    }
  }
  return db.article.findMany({
    where,
    include: { bezirke: { include: { bezirk: true } } },
    orderBy: [{ publishedAt: 'desc' }, { createdAt: 'desc' }],
    take: opts.limit ?? 50,
    skip: opts.offset ?? 0,
  })
}
```

## Open Questions

1. **HMAC cookie vs iron-session for Edge middleware**
   - What we know: Node `crypto` unavailable in Edge; Web Crypto `subtle` is available; iron-session (if installed) works with Web Crypto internally
   - What's unclear: Whether iron-session's overhead is justified for a single-editor tool
   - Recommendation: Hand-roll with `crypto.subtle` in middleware + `node:crypto` in Server Actions — zero additional dependency, sufficient security for single-editor use case

2. **`dead-man.ts` migration from env var to PipelineConfig**
   - What we know: Current implementation reads `process.env.DEAD_MAN_THRESHOLD_HOURS ?? '6'`; CONTEXT.md says PipelineConfig.deadManThresholdHours supersedes this
   - What's unclear: Whether to keep env var as fallback during Phase 5 transition
   - Recommendation: Read PipelineConfig from DB with a hard-coded fallback of `6` if row doesn't exist yet (same singleton pattern as AiConfig); remove env var dependency

3. **Article list performance with 200+ articles**
   - What we know: Success criterion requires finding any article in < 10 seconds; current `listArticles()` has DB indexes on `status`, `publishedAt`, `source + externalId`
   - What's unclear: Whether additional composite index on `(status, publishedAt)` or `(source, status)` is needed
   - Recommendation: Add `@@index([source, status])` and `@@index([isPinned, isFeatured])` in schema migration; Prisma filter queries will use these
